AD Recon

Active Directory reconnaissance is the systematic, pre-exploitation discovery of an AD environment's logical layout, trust relationships, privilege distrbution, and policy enforcement points. Its purpose is to transform an anonymous foothold-or an unprivileged domain account-into a navigable map that highlights the shortest path to high-value objectives such as Tier 0 assets, sensitive file shares, or cloud synchronization pipelines.


Unlike generic network scanning, AD recon abuses the very protocols the directory was built to serve: LDAP, DNS, MS-RPC, SAM-R, and Kerberos. A single LDAP query such as (objectCategory=group)(adminCount=1) immediately reveals all "protected" groups whose members are exempt from default ACL inheritance. A follow-up query against msDS-AllowedToActOnBehalfOfOtherIdentity surfaces Resource-Based Constrained Delegation (RBCD) attack paths without ever touching a workstation. DNS wildcard enumeration (adidnsdump) exposes forgotten test records that point to decommissioned-but still domain-joined-hosts ideal for relay attacks. Kerberos itself leaks intelligence: requesting a TGS (Ticket Granting Service) ticket for every observed SPN (Service Principal Name)GetUserSPNs.py) yields a target list for offline TGS cracking and identifies service accounts that may possess elevated privileges.



Credential hygiene reconnaissance focuses on password policy inference (Get-DomainPolicy), historical password reuse (msDS-KeyCredentialLink), and accounts marked "Password never expires" or "Reversible encryption required." BloodHound (or the SharpHound ingestor) automates much of this by performing LDAP walks, local group enumeration, and remote SAM-R queries, then building a directed graph where nodes are users, groups, and computers, and edges model administrative rights, session placement, and delegation rights. The resulting path queries (e.g., "Shortest path from Domain Users to Domain Admins") convert abstract ACLs (Access Control Lists) into an actual attack ininerary.



Recon also extends to trust topology. Command-line utilities such as nltest/domain_trusts and PowerShell's Get-ADTrust reveal forest, external, and shortcut trusts, their transivity, and whether SID (Session ID) filtering is enabled. Cross-forest SPN (Service Principal Name)Get-DomainUser-SPN-Domain target.com) can identify service accounts in a partner forest that are reachable from the initially compromised domain, paving the way for inter-forest Kerberos delegation attacks.



Operational security (OPSEC) considerations dictate stealth: LDAP queries are batched with page sizes ≤ 500, SAM-R enumeration is throttled to one RID (Relative ID) at a time, and Kerberos pre-authentication is requested only for accounts that are already known to exist (avoiding Event ID 4768 "Kerberos pre-authentication failed" storms). When performed correctly, AD recon leaves minimal forensic aritfacts while providing the adversary-or the red team operator-with a complete blueprint of identity, privilege, and policy that dictates every subsequent move toward domain dominance.


PowerShell Scan

Port Scan

<Import-Module Invoke-Portscan.ps1>
<#">
Invoke-Portscan -Hosts "websrv.domain.local,wsus.domain.local,apps.domain.local" -TopPorts 50 echo websrv.domain.local | Invoke-Portscan -oG test.gnmap -f -ports "80,443,8080" Invoke-Portscan -Hosts 172.16.0.0/24 -T 4 -TopPorts 25 -oA localnet #>

The above snippet is a minimal usage guide for the PowerShell port-scanning module Invoke-Portscan.ps1, a native-code wrapper around the open-source pscan engine that ships with the PowerSploit post-exploitation framework. In effect, it is a command-line TCP/UDP scanner that runs entirely in-process, requires no administrative privileges on the scanning host, and outputs results in formats compatible with nmap (-oG, -oA) so they can be fed into other tools such as Metasploit, EyeWitness, or BloodHound's session-collection ingestors.

  1. Import-Module Invoke-Portscan.ps1 Loads the module into the current run-space. The file contains a compiled C# binary (pscan.dll) that is reflectively injected, giving the attacker a fast, low-profile scanner without dropping nmap.exe to disk.
  2. Comment block <# ... #>

    Demonstrates three common scan patterns:

    a. Invoke-Portscan -Hosts "websrv.domain.local,wsus.domain.local,apps.domain.local" -TopPorts 50
    Resolves the three supplied FQDNs (Fully Qualified Domain Names), then probes each IP for the top 50 most frequently open TCP ports (as ranked by nmap-services), Output is printed to the console in a human-readable table.

    b. echo websrv.domain.local | Invoke-Portscan -oG test.gnmap -f ports "80,443,8080"
    Pipes a single hostname into the scanner, forces -f a full TCP three-way handshake on ports 80, 443, and 8080, and writes grep-able (-oG) output to test.gnmap. The file can later be parsed with grep or imported into Metasploit's db_import.

    c. Invoke-Portscan -Hosts 172.16.0.0/24 -T 4- TopPorts 25 -oA localnet
    Scans the entire /24 subnet using timing template 4 (aggressive: ~1000 packets/second), tests the top 25 TCP ports, and saves results in "all formats" (-oA):localnet.gnmap,localnet.xml, and localnet.txt. This is ideal for quickly mapping a newly compromised LAN segment before pivoting deeper into the network abyss.

    In a red team context, the module is typically executed from a non-privleged PowerShell session on a compromised workstation. Because it never calls CreateProcess on nmap.exe, it avoids common application-allowlisting rules and leaves a minimal forensic footprint-only a .NET assembly loaded in memory and a handful of outbound TCP SYN packets that blend into normal workstation network traffic. The resulting port lists are then cross-referenced with BloodHound's session data to identify file servers, WSUS boxes, or SQL listeners that may provide next-hop credentials or relay opportunities.

Active Directory Without RSAT

Running the Active Directory PowerShell module normally requires the Remote Server Administration Tools (RSAT) package to be installed. RSAT drops not only the dozen .psm1/.ps1 XML help files but also the 100%-managed assembly Microsoft.ActiveDirectory.Management.dll (plus its satellite *.resources.dll). That single DLL contains every cmdlet (Get-ADUser,Get-ADGroup,Get-ADComputer) and the AD Web Service (ADWS) client stack that knows how to speak the WS-Transfer protocol to a domain controller's ADWS endpoint (port 9389).

On hardened workstations or during red team engagements you rarely have the luxury of running Add-WindowsFeature RSAT - it needs elevated rights, reboots, or Internet access. The workaround is to sideload only the bare minimum:

1. Build (or borrow) a throw-away Windows VM, join it to any AD domain, and install RSAT through the normal GUI or with Enable-WindowsOptionalFeature -Online- FeatureName RSATClient-Ad-PowerShell
The installer places the 64-bit assembly in C:\Windows\Microsoft.NET\assembly\GAC_64\Microsoft.ActiveDirectory.Management\
Copy the folder contents-the main DLL and is language sub-folder-to your attack box (USB, encoded gzip, any other method).

2. On the target workstation drop the DLL into any writeable directory, e.g.,:C:\Users\Public\AD\Microsoft.ActiveDirectory.Management.dll

3. Import the assembly directly-no RSAT, no manifest, no registry keys required: Import-Module .\Microsoft.ActiveDirectory.Management.dll
PowerShell loads the assembly into the current AppDomain and auto-registers every cmdlet exported by CompiledCmdlet attributes inside the DLL.

4. Proving it works: Get-Command get-adcom* should return cmdlets such as Get-ADComputer,Get-ADUser,Get-ADGroup, etc.
If you see them, the module is live; you can now run Get-ADUser -Filter* -Properties MemberOf,PasswordLastSet or any other AD cmdlet as long as the machine can reach a domain controller on TCP 9389 (ADWS) or TCP 445 (SMB for legacy ADSI) and you possess valid credentials (domain-joined session, runas /netonly, or explicit -Credential.

Caveats & OPSEC Notes

  • No help files are present, so Get-Help Get-ADUser will fail unless you also copy the about_* text files, functionality remains intact.
  • The DLL is pure managed code-no kernel drivers-so it bypasses most application allowlisting rules that only block unmanaged EXEs.
  • All network traffic is signed/sealed under the authenticated user's context: use runas /netonly or New-PSSession-Authentication CredSSP when operating from an off-domain jump box.
  • If ADWS is disabled on DCs (rare), fall back to ADSI via ([ADSI]"LDAP://DC=corp,DC=local") - but that path requires 445/139 and is slower.

In short, extracting and importing only Microsoft.ActiveDirectory.Management.dll gives you the full AD PowerShell arsenal without every having to install RSAT, leaving a minimal forensic footprint and zero need for local administrator rights on the scanning workstation.

General Functions of PowerView

Miscellaneous Functions:

Below is a concise technician's notebook for each of the PowerView/PowerSploit helper functions. Every description states what the cmdlet actually returns or modifies, the protocol/Win32 API it abuses, and the single most common offensive use-cases.

1. Export-PowerViewCSV # Thread-safe wrapper around [System.IO.File]::AppendAllText(). Output objects are serialized with ConvertTo-Csv -NoTypeInformation and written under a Mutex so parallel threads (e.g., -Threads20 in Invoke-ShareFinder) does not corrupt the file. Attack Value: Keeps distributed recon results in one mergeable file for ingestion into the BloodHound tool. 2. Set-MacAttribute # Uses SetFileTime,SetFileAttributes, and DeviceIoControl (FSCTL_SET_OBJECT_ID) to clone MAC (Modified-Access-Created) timestamps, the 8-byte object-ID, and even the 16-byte birth-volume ID from a donor file. Attack Value: Timestomps implants so that they match a legitimate SYSVOL DLL or a user's template document. 3. Copy-ClonedFile # Wrapper around CopyFileEx + Set-MacAttribute. Copies a payload to \\target\C$\Temp while keeping size, timestamps, and object-ID identical to the source. Attack Value: Drops an executable that evades "new file" SIEM rules or time-ranged triage. 4. Get-IPAddress # Simple [System.Net.Dns]::GetHostEntry($Name). If the host has Link-Local Multicast Name Resolution (LLMNR) disabled it falls back to NBNS. Attack Value: Converts harvested server names from Shares.csv into pivot targets without calling nslookup. 5. Test-Server # Tries TCP445, 135, or the port you supply via -Port. Uses System.Net.Sockets.TcpClient with a 1-second timeout by default. Attack Value: Fast up/down filter before running Invoke-SMBClient or Invoke-WMICheck. 6. Convert-NameToSid # Calls LookupAccountName against the remote LSA (Local Security Authority) policy database. Returns both the binary SID and the string SID. Attack Value: Turns "DEV\SQL-Service" into S-1-5-21-...-1104 for Silver Ticket forging. 7. Convert-SidToName # Inverse: LookupAccountSid. Useful for when all you have is a RID from Get-DomainGroupMember. Attack Value: Deobfuscates the human name behind a privileged SID discovered in an ACL. 8. Convert-NT4toCanonical # Uses DsCrackNames against the AD Global Catalog to translate DOMAIN\user to domain.com/Users/User. Attack Value: Feeds canonical names to Get-ObjectAcl without string-splitting logic. 9. Get-Proxy # Reads per-machine proxy settings from HKLM:\SOFTWARE\Policies\Microsoft\Windows\CurrentVersion\Internet Settings and per-user from HKCU:\... . Also honors WPAD/PAC URLs. Attack Value: Tells you which egress proxy to tunnel through before launching Invoke-WebRequest for payload download. 10. Get-PathAcl # Enumerates the security descriptor on a local or UNC (Universal Naming Convention) path via GetNamedSecurityInfo. Recursively expands group membership so you see every principal that can modify the file. Attack Value: Finds C:\ProgramData\Vendor\Update.exe that Authenticated Users can overwrite-privilege escalation. 11. Get-UserProperty # Note: You can also use Get-ComputerProperty. Generic LDAP search that can dump any attribute (or a comma-separated list) for every user / computer in the domain. Supports -Properties* or targeted properties like msDS-AllowedToDelegateTo.
