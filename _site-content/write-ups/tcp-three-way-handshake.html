The Fatal Flaw: Exploiting the TCP Three-Way Handshake

 The SYN flood attack is a classic demonstration of exploiting a fundamental, necessary mechanism of the Internet: the TCP Three-Way Handshake. To fully appreciate the attack, one must understand the standard connection process first.

The Standard Connection Choreography

For a reliable connection to be established between a client and a server, TCP (Transmission Control Protocol) requires a three-step exchange.

  1. SYN (Synchronize): A client sends a SYN packet to the server, initiating the desire to communicate.

  2. SYN-ACK (Synchronize-Acknowledge): The server receives the request, sends a SYN-ACK back to the client, and critically, it allocates resources for the prospective connection. This includes space in a backlog queue and memory for connection state information. At this point, the connection is in a "half-open" state.

  3. ACK (Acknowledge): The client sends a final ACK packet, confirming the receipt of the server's acknowledgment. The connection is now established, and data transfer can begin.

The Attack: Flooding the Backlog Queue

The SYN flood attack specifically targets step 2, the resource allocation phase. The attacker sends a massive volume of initial SYN packets but deliberately never sends the final ACK.

This is often achieved by IP address spoofing, where the attacker disguises the source IP address of the SYN packets. When the target server sends its SYN-ACK response, it goes to a non-existent or unreachable address. Because there is no legitimate client to complete the connection, the half-open connection sits dormant, consuming the allocated resources in the server's backlog queue.

Consequence: Resource Exhaustion and Denial of Service (DoS)

Server operating systems maintain a finite-sized data structure (the backlog queue) to track these half-open connection requests. As the attacker continues to send thousands of incomplete SYN packets, this queue quickly becomes saturated as there are too many null processes memory has to handle.

Once the backlog queue is full, the server is physically unable to allocate resources for any new or incoming connection attempts. Any subsequent SYN packets, even those coming from legitimate sources or users, are simply dropped. The server remains online, but its capacity to accept new connections is completely choked. Legitimate sessions time out, critical services degrade, and the target ultimately crashes at some point, is effectively taken offline, thus achieving the primary goal of the Denial of Service (DoS) attack.
